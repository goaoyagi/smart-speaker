# AGENTS.md — Repository-wide Rules for AI Coding Agents

このファイルは、`goaoyagi/smart-speaker` リポジトリで作業する AI エージェント向けの共通ルールです。  
全セッションで一貫して守ること。

## 1. Language / Communication

- すべてのユーザー向け回答は **日本語** で行う。
- 回答は先に結論を述べ、その後に必要な補足を書く。
- 不確実な内容は断定せず、前提・不確実性・確認方法を明示する。

## 2. Safety / Secrets

- 秘密情報（APIキー、トークン、秘密鍵、認証情報）を生成・出力・コミットしない。
- `.env` 等の秘匿ファイルを新規追加・変更しない（必要な場合は `.env.example` のみ）。
- ログやエラーメッセージに秘匿情報が含まれる可能性がある場合はマスクする。

## 3. Scope of Changes

- 指示範囲外のリファクタや無関係な修正を混ぜない。
- 既存挙動を変える場合は、**理由・影響範囲・互換性** を説明する。
- 破壊的変更（Breaking Change）は、明示的な指示がある場合のみ行う。

## 4. Code Quality

- 既存の設計・命名・レイヤ構成・コーディング規約を優先して踏襲する。
- 変更は最小差分を基本とし、読みやすさと保守性を優先する。
- マジックナンバーや不明瞭な分岐には、必要に応じて定数化やコメントを付ける。

## 5. Testing

- 変更箇所に対するテスト（または検証手順）を必ず提示する。
- テスト追加が難しい場合は、理由と代替の確認手順（再現手順、手動確認観点）を記載する。
- 既存テストがある領域では、可能な限りテストを追加・更新して回帰を防ぐ。

## 6. Output Format for Responses

変更提案時は、原則として次の順で出力する：

1. **結論（1〜2行）**
2. **変更内容（箇条書き）**
3. **理由（なぜその変更か）**
4. **確認方法（テスト手順・実行コマンド）**
5. **リスク/未対応（あれば）**

## 7. Pull Request Guidelines

PR本文には最低限、以下を含める：

- 目的 / 背景
- 変更内容の要約
- テスト結果（実行コマンド・結果）
- 影響範囲（ユーザー影響、互換性、運用影響）
- 懸念点・フォローアップ（あれば）

## 8. Commit Guidelines

- コミットメッセージは簡潔で意味が分かる単位にする。
- 可能なら Conventional Commits を推奨（`feat:`, `fix:`, `chore:`, `docs:` など）。
- 大きすぎる変更は論理的な単位に分割する。

## 9. Prohibited Actions (Without Explicit Instruction)

明示的な依頼なしに以下を行わない：

- 依存パッケージの大規模アップデート
- CI/CD 設定の大幅変更
- 本番設定値の変更
- ライセンス/法務に関わる記述変更
- データ削除を伴うマイグレーション

## 10. When Requirements Are Ambiguous

- まず安全側の実装を提案する。
- 要件が複数解釈できる場合、解釈A/Bと推奨案を短く提示する。
- 追加確認が必要な点は、具体的な質問として列挙する。

---

このルールに反する上位指示（ユーザーの明示指示）がある場合は、  
その指示を優先しつつ、想定リスクを説明した上で実行すること。
